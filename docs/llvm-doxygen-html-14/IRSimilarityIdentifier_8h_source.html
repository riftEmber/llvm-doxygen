9// \file
10// Interface file for the IRSimilarityIdentifier for identifying similarities in
11// IR including the IRInstructionMapper, which maps an Instruction to unsigned
12// integers.
13//
14// Two sequences of instructions are called "similar" if they perform the same
15// series of operations for all inputs.
16//
17// \code
18// %1 = add i32 %a, 10
19// %2 = add i32 %a, %1
20// %3 = icmp slt icmp %1, %2
21// \endcode
22//
23// and
24//
25// \code
26// %1 = add i32 11, %a
27// %2 = sub i32 %a, %1
28// %3 = icmp sgt icmp %2, %1
29// \endcode
30//
31// ultimately have the same result, even if the inputs, and structure are
32// slightly different.
33//
34// For instructions, we do not worry about operands that do not have fixed
35// semantic meaning to the program. We consider the opcode that the instruction
36// has, the types, parameters, and extra information such as the function name,
37// or comparison predicate. These are used to create a hash to map instructions
38// to integers to be used in similarity matching in sequences of instructions
39//
40// Terminology:
41// An IRSimilarityCandidate is a region of IRInstructionData (wrapped
42// Instructions), usually used to denote a region of similarity has been found.
43//
44// A SimilarityGroup is a set of IRSimilarityCandidates that are structurally
45// similar to one another.
46//
47//===----------------------------------------------------------------------===//

77/// This provides the utilities for hashing an Instruction to an unsigned
78/// integer. Two IRInstructionDatas produce the same hash value when their
79/// underlying Instructions perform the same operation (even if they don't have
80/// the same input operands.)
81/// As a more concrete example, consider the following:
82///
83/// \code
84/// %add1 = add i32 %a, %b
85/// %add2 = add i32 %c, %d
86/// %add3 = add i64 %e, %f
87/// \endcode
88///
89// Then the IRInstructionData wrappers for these Instructions may be hashed like
90/// so:
91///
92/// \code
93/// ; These two adds have the same types and operand types, so they hash to the
94/// ; same number.
95/// %add1 = add i32 %a, %b ; Hash: 1
96/// %add2 = add i32 %c, %d ; Hash: 1
97/// ; This add produces an i64. This differentiates it from %add1 and %add2. So,
98/// ; it hashes to a different number.
99/// %add3 = add i64 %e, %f; Hash: 2
100/// \endcode
101///
102///
103/// This hashing scheme will be used to represent the program as a very long
104/// string. This string can then be placed in a data structure which can be used
105/// for similarity queries.
106///
107/// TODO: Handle types of Instructions which can be equal even with different
108/// operands. (E.g. comparisons with swapped predicates.)
109/// TODO: Handle CallInsts, which are only checked for function type
110/// by \ref isSameOperationAs.
111/// TODO: Handle GetElementPtrInsts, as some of the operands have to be the
112/// exact same, and some do not.

121 /// and is used when checking when two instructions are considered similar.

126 /// This is only relevant if we are wrapping a CmpInst where we needed to

128 /// to a less than form. It is None otherwise.
129 Optional<CmpInst::Predicate> RevisedPredicate;

131 /// This is only relevant if we are wrapping a CallInst. If we are requiring
132 /// that the function calls have matching names as well as types, and the
133 /// call is not an indirect call, this will hold the name of the function. If
134 /// it is an indirect string, it will be the empty string. However, if this

136 /// function call type. The value held here is used to create the hash of the
137 /// instruction, and check to make sure two instructions are close to one

141 /// This structure holds the distances of how far "ahead of" or "behind" the
142 /// target blocks of a branch, or the incoming blocks of a phi nodes are.

145 /// Code Example:
146 /// \code
147 /// block_1:

149 /// block_2:

154 /// block_1:
155 /// br i1 %0, distance 1, distance 2
156 /// block_2:
157 /// br i1 %1, distance -1, distance 0
158 /// block_3:
159 /// br i1 %2, distance -1, distance -2

162 /// "ahead" of block_2.

227 /// Hashes \p Value based on its opcode, types, and operand types.
228 /// Two IRInstructionData instances produce the same hash when they perform
229 /// the same operation.
230 ///
231 /// As a simple example, consider the following instructions.
232 ///
233 /// \code
234 /// %add1 = add i32 %x1, %y1
235 /// %add2 = add i32 %x2, %y2
236 ///
237 /// %sub = sub i32 %x1, %y1
238 ///
239 /// %add_i64 = add i64 %x2, %y2
240 /// \endcode
241 ///
242 /// Because the first two adds operate the same types, and are performing the
243 /// same action, they will be hashed to the same value.
244 ///
245 /// However, the subtraction instruction is not the same as an addition, and
246 /// will be hashed to a different value.
247 ///
248 /// Finally, the last add has a different type compared to the first two add
249 /// instructions, so it will also be hashed to a different value that any of
250 /// the previous instructions.
251 ///
252 /// \param [in] ID - The IRInstructionData instance to be hashed.
253 /// \returns A hash_value of the IRInstructionData.
254 friend hash_code hash_value(const IRInstructionData &ID) {
255 SmallVector<Type *, 4> OperTypes;
256 for (Value *V : ID.OperVals)
257 OperTypes.push_back(V->getType());
258
259 if (isa<CmpInst>(ID.Inst))
260 return llvm::hash_combine(
261 llvm::hash_value(ID.Inst->getOpcode()),
262 llvm::hash_value(ID.Inst->getType()),
263 llvm::hash_value(ID.getPredicate()),
264 llvm::hash_combine_range(OperTypes.begin(), OperTypes.end()));
265
266 if (IntrinsicInst *II = dyn_cast<IntrinsicInst>(ID.Inst)) {
267 // To hash intrinsics, we use the opcode, and types like the other
268 // instructions, but also, the Intrinsic ID, and the Name of the
269 // intrinsic.
270 Intrinsic::ID IntrinsicID = II->getIntrinsicID();
271 return llvm::hash_combine(
272 llvm::hash_value(ID.Inst->getOpcode()),
273 llvm::hash_value(ID.Inst->getType()), llvm::hash_value(IntrinsicID),
274 llvm::hash_value(*ID.CalleeName),
275 llvm::hash_combine_range(OperTypes.begin(), OperTypes.end()));
276 }
277
278 if (isa<CallInst>(ID.Inst)) {
279 std::string FunctionName = *ID.CalleeName;
280 return llvm::hash_combine(
281 llvm::hash_value(ID.Inst->getOpcode()),
282 llvm::hash_value(ID.Inst->getType()),
283 llvm::hash_value(ID.Inst->getType()), llvm::hash_value(FunctionName),
284 llvm::hash_combine_range(OperTypes.begin(), OperTypes.end()));
285 }
286
287 return llvm::hash_combine(
288 llvm::hash_value(ID.Inst->getOpcode()),
289 llvm::hash_value(ID.Inst->getType()),
290 llvm::hash_combine_range(OperTypes.begin(), OperTypes.end()));
291 }

299/// Compare one IRInstructionData class to another IRInstructionData class for
300/// whether they are performing a the same operation, and can mapped to the
301/// same value. For regular instructions if the hash value is the same, then
302/// they will also be close.
303///
304/// \param A - The first IRInstructionData class to compare
305/// \param B - The second IRInstructionData class to compare
306/// \returns true if \p A and \p B are similar enough to be mapped to the same
307/// value.
308bool isClose(const IRInstructionData &A, const IRInstructionData &B);

310struct IRInstructionDataTraits : DenseMapInfo<IRInstructionData *> {
311 static inline IRInstructionData *getEmptyKey() { return nullptr; }
312 static inline IRInstructionData *getTombstoneKey() {
313 return reinterpret_cast<IRInstructionData *>(-1);
314 }
315
316 static unsigned getHashValue(const IRInstructionData *E) {
317 using llvm::hash_value;
318 assert(E && "IRInstructionData is a nullptr?");
319 return hash_value(*E);
320 }
321
322 static bool isEqual(const IRInstructionData *LHS,
323 const IRInstructionData *RHS) {
324 if (RHS == getEmptyKey() || RHS == getTombstoneKey() ||
325 LHS == getEmptyKey() || LHS == getTombstoneKey())
326 return LHS == RHS;
327
328 assert(LHS && RHS && "nullptr should have been caught by getEmptyKey?");
329 return isClose(*LHS, *RHS);
330 }

333/// Helper struct for converting the Instructions in a Module into a vector of
334/// unsigned integers. This vector of unsigned integers can be thought of as a
335/// "numeric string". This numeric string can then be queried by, for example,
336/// data structures that find repeated substrings.
337///
338/// This hashing is done per BasicBlock in the module. To hash Instructions
339/// based off of their operations, each Instruction is wrapped in an
340/// IRInstructionData struct. The unsigned integer for an IRInstructionData
341/// depends on:
342/// - The hash provided by the IRInstructionData.
343/// - Which member of InstrType the IRInstructionData is classified as.
344// See InstrType for more details on the possible classifications, and how they
345// manifest in the numeric string.
346///
347/// The numeric string for an individual BasicBlock is terminated by an unique
348/// unsigned integer. This prevents data structures which rely on repetition
349/// from matching across BasicBlocks. (For example, the SuffixTree.)
350/// As a concrete example, if we have the following two BasicBlocks:
351/// \code
352/// bb0:
353/// %add1 = add i32 %a, %b
354/// %add2 = add i32 %c, %d
355/// %add3 = add i64 %e, %f
356/// bb1:
357/// %sub = sub i32 %c, %d
358/// \endcode
359/// We may hash the Instructions like this (via IRInstructionData):
360/// \code
361/// bb0:
362/// %add1 = add i32 %a, %b ; Hash: 1
363/// %add2 = add i32 %c, %d; Hash: 1
364/// %add3 = add i64 %e, %f; Hash: 2
365/// bb1:
366/// %sub = sub i32 %c, %d; Hash: 3
367/// %add4 = add i32 %c, %d ; Hash: 1
368/// \endcode
369/// And produce a "numeric string representation" like so:
370/// 1, 1, 2, unique_integer_1, 3, 1, unique_integer_2
371///
372/// TODO: This is very similar to the MachineOutliner, and should be
373/// consolidated into the same interface.

375 /// The starting illegal instruction number to map to.
376 ///
377 /// Set to -3 for compatibility with DenseMapInfo<unsigned>.
378 unsigned IllegalInstrNumber = static_cast<unsigned>(-3);

380 /// The next available integer to assign to a legal Instruction to.
381 unsigned LegalInstrNumber = 0;
382

589/// This is a class that wraps a range of IRInstructionData from one point to
590/// another in the vector of IRInstructionData, which is a region of the
591/// program. It is also responsible for defining the structure within this
592/// region of instructions.
593///
594/// The structure of a region is defined through a value numbering system
595/// assigned to each unique value in a region at the creation of the
596/// IRSimilarityCandidate.
597///
598/// For example, for each Instruction we add a mapping for each new
599/// value seen in that Instruction.
600/// IR: Mapping Added:
601/// %add1 = add i32 %a, c1 %add1 -> 3, %a -> 1, c1 -> 2
602/// %add2 = add i32 %a, %1 %add2 -> 4
603/// %add3 = add i32 c2, c1 %add3 -> 6, c2 -> 5
604///
605/// We can compare IRSimilarityCandidates against one another.
606/// The \ref isSimilar function compares each IRInstructionData against one
607/// another and if we have the same sequences of IRInstructionData that would
608/// create the same hash, we have similar IRSimilarityCandidates.
609///
610/// We can also compare the structure of IRSimilarityCandidates. If we can
611/// create a mapping of registers in the region contained by one
612/// IRSimilarityCandidate to the region contained by different
613/// IRSimilarityCandidate, they can be considered structurally similar.
614///
615/// IRSimilarityCandidate1: IRSimilarityCandidate2:
616/// %add1 = add i32 %a, %b %add1 = add i32 %d, %e
617/// %add2 = add i32 %a, %c %add2 = add i32 %d, %f
618/// %add3 = add i32 c1, c2 %add3 = add i32 c3, c4
619///
